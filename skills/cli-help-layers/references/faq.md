# 问答

**设计理念来源？**
- Linux 传统 CLI 规范，参数风格、子命令结构、man 手册等
- tldr 的用例（case）设计方式，以示例驱动的简明文档
- 现有 skills 设计规范，如：能力抽象、参数化调用、渐进式披露等

**为什么用 @USAGE / @HELP / @MAN 作为区块标记？**
Agent 可以用一行正则 `^@(USAGE|HELP|MAN)\s+(.+)$` 精确定位到需要的信息层级，不需要理解自然语言上下文。区块内的大写标签（如 OPTIONS:、ENUMS:）同理，都是为了让 LLM 做结构化抽取的成本降到最低。

**为什么 usage 只放 3-5 个示例而不是更多？**
参考 tldr 的实践 —— 信息越少越容易被完整消化。agent 的核心能力是从少量示例归纳模式（in-context learning），5 个精选示例比 20 个罗列示例更有效。

**为什么错误信息需要四行结构（code + name + trigger + fix）？**
agent 遇到错误后需要的不是"理解错误"而是"知道怎么修"。fix 字段直接给出修复动作，example 字段给出修复后的完整命令，agent 可以直接重试而无需推理。
