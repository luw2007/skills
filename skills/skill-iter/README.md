# Skill Iter

> Skill 自迭代评审引擎 — 评估并提升 Skill 的自迭代闭环能力

## 这是什么

Skill Iter 解决一个核心问题：**Skill 执行完一次后，下次执行能自动变好吗？**

大多数 Skill 是静态的 — 写好就不再进化。本 skill 定义了 **7 个维度的自迭代能力评审标准**，配套完整工具链，形成"评审 → 分析 → 改进 → 验证"闭环。

## 自迭代三层能力模型

| 层级 | 名称 | 本质 |
|------|------|------|
| L1 | 运行时自修复 | 边执行边修正过时步骤 |
| L2 | 执行后经验沉淀 | 每次执行的教训自动积累 |
| L3 | 跨 Skill 知识联动 | 一个 skill 的经验惠及其他 skill |

## 7 维度评审标准

| 维度 | 评审内容 | 核心问题 |
|------|----------|----------|
| D1 反馈采集 | 执行结果状态、轮数、错误 | 有数据进来吗？ |
| D2 触发判定 | 失败 + 阻塞 + "成功但低效" | 什么时候值得分析？ |
| D3 分析能力 | 三分法：固化/经验/忽略 | 能提取可操作的改进吗？ |
| D4 经验持久化 | Lessons Learned + 去重 + 淘汰 | 分析结果写到哪里？ |
| D5 注入闭环 | 下次执行自动读到新经验 | 经验能回到下次执行吗？ |
| D6 安全门禁 | .pending 暂存 + prompt 注入检测 | 自迭代会引入破坏吗？ |
| D7 可观测性 | 触发原因 + session_id + 时间戳 | 能追溯迭代历史吗？ |

## 评审报告示例

```
📊 自迭代能力评审报告 — mirror-release

| 维度 | 等级 | 说明 |
|------|------|------|
| D1 反馈采集 | ✅ | Phase 14 采集错误/耗时/阻塞 |
| D2 触发判定 | ⚠️ | 每次都执行，未区分"值得分析" |
| D3 分析能力 | ✅ | 三分法（固化/规避/忽略） |
| D4 经验持久化 | ⚠️ | 有 Lessons Learned 但无淘汰策略 |
| D5 注入闭环 | ✅ | Tier 2 加载自动包含 |
| D6 安全门禁 | ⚠️ | 依赖 git，无 .pending |
| D7 可观测性 | ⚠️ | 仅 git log |

总评：✅ 达标 3/7 | ⚠️ 基础 4/7
```

## 工具链

| 脚本 | 功能 | 对应维度 |
|------|------|----------|
| `collect_feedback.py` | 采集执行反馈 | D1 |
| `should_improve.py` | 判定是否触发改进 | D2 |
| `analyze_trajectory.py` | 三分法分析 | D3 |
| `merge_lessons.py` | 管理 Lessons Learned | D4, D6 |
| `threat_scan.py` | Prompt 注入检测 | D6 |
| `run_retrospective.py` | 端到端编排管线 | D5, D7 |

闭环路径：`Agent 读取 SKILL.md → 执行评审 → 采集反馈 → 分析改进 → 写入 .pending → 人工确认合入 → 下次读取时自动包含`
