# 探索树详细格式规范

## 探索树 Markdown 格式

```markdown
# 探索日志: [任务名称]

## 元信息
- 目标: [最终目标，一句话]
- 约束: [不可违反的条件]
- 已确认事实: [编号列表]

## 探索树

### 第 N 层（广度枚举 → YYYY-MM-DD）

当前层问题: [此层要解决的核心子问题]
策略: [选择依据]

#### N.1 [路径名] [难度] [状态]

| 假设 | 一句话核心假设 |
| 方法 | 验证手段 |
| 预期 | 假设成立应观察到什么 |
| 实际 | 实际观察 |
| 结论 | 成功/失败原因 |
| 证据 | 关键数据（精简） |
| 排除 | 原理不通 / 实现障碍 / 不满足约束 |

  ##### N.1.1 [子路径] [状态]  ← 深度展开
  ##### N.1.2 [子路径] [状态]

#### N.2 [路径名] [难度] [状态]
#### N.3 [路径名] [难度] [状态]

---
### 反思 R-[序号] (层N → 层N+1)
...
```

## 状态标签与难度

**状态**: ✅ 成功 | ❌ 失败 | ⏸️ 阻塞 | 🔄 深度中 | 💡 待浅探 | ⏭️ 跳过

**难度**: ⭐ 容易(< 10min) | ⭐⭐ 中等(< 1h) | ⭐⭐⭐ 困难(> 1h)

**排除分类**:
- **原理不通** — 假设本身被证伪，此路永远不通
- **实现障碍** — 理论可行但当前条件不具备
- **不满足约束** — 方案有效但违反项目约束

## 状态文件格式 (`<task-name>.state.json`)

每次节点状态变更时，**同步更新** `state.json`。这是探索树的 single source of truth。

### JSON Schema

```json
{
  "task": "任务名",
  "goal": "最终目标，一句话",
  "constraints": ["约束1", "约束2"],
  "confirmed_facts": ["已确认事实1", "已确认事实2"],
  "current_layer": 2,
  "focus": "2.1",
  "strategy": "当前策略说明",
  "stats": { "success": 1, "failed": 4, "active": 1, "pending": 3, "blocked": 0, "skipped": 0 },
  "layers": [
    {
      "id": 1,
      "question": "此层要解决的核心子问题",
      "strategy": "选择依据",
      "created_at": "2026-03-15T10:00:00Z",
      "nodes": [
        {
          "id": "1.1",
          "name": "方案名",
          "difficulty": 1,
          "status": "failed",
          "hypothesis": "核心假设",
          "method": "验证手段",
          "expected": "预期观察",
          "actual": "实际观察",
          "conclusion": "结论",
          "evidence": "关键证据",
          "exclusion": { "type": "principle|implementation|constraint", "reason": "排除原因" },
          "updated_at": "2026-03-15T10:05:00Z",
          "children": []
        }
      ]
    }
  ],
  "reflections": [
    {
      "id": "R-01",
      "from_layer": 1,
      "to_layer": 2,
      "excluded": ["1.1", "1.2", "1.4"],
      "common_failure_pattern": "描述",
      "overlooked_assumptions": ["假设1"],
      "new_paths": [
        { "name": "路径名", "hypothesis": "假设", "difficulty": 1 }
      ],
      "created_at": "2026-03-15T11:00:00Z"
    }
  ],
  "updated_at": "2026-03-15T12:00:00Z"
}
```

**字段说明**：
- `difficulty`: 整数 1-3，对应 ⭐/⭐⭐/⭐⭐⭐
- `status`: `"success"` | `"failed"` | `"blocked"` | `"active"` | `"pending"` | `"skipped"`
- `exclusion.type`: `"principle"`(原理不通) | `"implementation"`(实现障碍) | `"constraint"`(不满足约束)
- `children`: 递归同结构节点数组（深度展开）

## 反思模板

当一层耗尽时触发：

```
### 反思 R-[序号] (层N → 层N+1)

- 本层排除: [列表]
- 共同失败模式: [有无重复原因？]
- 被忽视的假设: [哪些"想当然"可能是错的？]
- 未覆盖攻击面:
  1. ...
  2. ...
- 新层路径:
  1. [名称] — [假设] — [难度]
  2. ...
```

## 并发控制（`.lock` + timeout）

多个 agent session 可能同时对同一探索树操作。通过文件锁防止并发写入冲突。

### Lock 文件格式 (`<task-name>.lock`)

```json
{
  "session_id": "当前 agent session 标识",
  "pid": 12345,
  "acquired_at": "2026-03-15T10:00:00Z",
  "timeout_seconds": 300
}
```

### 协议规则

1. **获取锁**：操作 `state.json` 或 `.md` 之前，检查 `.lock` 文件
   - 不存在 → 创建 `.lock`，写入当前 session 信息
   - 已存在 → 检查是否超时

2. **超时判定**：`now - acquired_at > timeout_seconds` 则视为死锁
   - 超时 → 删除旧锁，创建新锁，继续操作
   - 未超时 → **等待或放弃**，输出提示：`"探索树被 session {id} 锁定，{remaining}s 后超时"`

3. **释放锁**：完成 `state.json` 写入后，立即删除 `.lock` 文件

4. **默认超时**：300 秒（5 分钟）。单次深度操作预计不超过此时长

5. **续租**：长时间操作中，每 60 秒更新 `acquired_at` 续租（防止被误判超时）

### 操作流程

```
[开始操作]
    │
    ├─ .lock 不存在？ ──→ 创建 .lock ──→ 读取/更新 state.json + .md ──→ 删除 .lock
    │
    └─ .lock 已存在？
        ├─ 已超时？ ──→ 覆盖 .lock ──→ 继续操作
        └─ 未超时？ ──→ 输出等待提示，不操作
```
